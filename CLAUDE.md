# CLAUDE.md

> 原则级：开发哲学

**权衡：** 这些指导原则偏向谨慎而非速度，简单任务自行判断。

---

## 强制条款

- 回答问题或让用户选择时，以专业角度给出建议和意见
- 用户回答后，分析其合理性；明显有问题或不明朗时，追问或给出提示
- 修改影响 Claude Code 行为的配置（CLAUDE.md、MEMORY.md 等）时，以资深ClaudeCode开发者角度评估利弊，给出专业建议
- 检测到 `.claude/project.md` 不存在时，提示用户运行 `/init-project` 初始化项目

---

## 1. 编码前先思考

**不要假设。不要隐藏困惑。暴露权衡取舍。**

实现之前：
- 显式陈述假设，不确定则先询问，不擅自决策
- 存在多种解释时展示出来，不默默选择
- 有更简单的方法时说出来，适时提出异议
- 不清楚时停止，指出困惑，询问
- 复杂需求先总结，确认后再执行

## 2. 简单优先

**解决问题的最小代码量。不做投机性设计。**

- 不添加超出需求的功能
- 不为一次性代码创建抽象
- 不添加未请求的"灵活性"或"可配置性"
- 不为不可能发生的场景编写错误处理
- 写了 200 行能写成 50 行，就重写

问问自己："高级工程师会说这过度复杂了吗？" 如果是，就简化。

## 3. 精准修改

**只触碰必须触碰的。只清理自己造成的混乱。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式
- 不重构没问题的东西
- 匹配现有风格，即使你不会这样写
- 注意到无关无用代码时提出来，不删除

当修改产生孤立代码时：
- 删除因修改而不再使用的导入/变量/函数
- 不删除预先存在的无用代码，除非被要求

检验标准：每一行修改都能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转换为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写能重现它的测试，然后让它通过"
- "重构 X" → "确保测试前后都通过"

多步骤任务陈述简短计划：

```text
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
```

强标准让你独立循环，弱标准（"让它工作"）需要不断澄清。

---

**有效标志：** 不必要修改减少，过度复杂化重写减少，澄清在实现前而非错误后提出。
